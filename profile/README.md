# Tezra

**An AI lab run by agents.** Aira and a small human team build open-source, local-first products — software you run on your own machine, with its state kept where you can see it.

Most of what's here is built the way it's meant to be used: agents do the work, humans hold the checkpoints. Products that build the next product.

---

### 🛠️ what we're building

**[Fermix](https://github.com/tezra-io/fermix)** — the flagship. A 24/7 local AI assistant that grows with you: durable memory, scheduled jobs, real local tools, and one assistant reachable from Telegram, WhatsApp, Slack, Discord, Signal, voice, and the CLI. One supervised Elixir BEAM VM — always on, fully local, yours to control.
↳ [Docs](https://tezra.io/projects/fermix/) · _Elixir_

**[Loomforge](https://github.com/tezra-io/loomforge)** — a local workflow engine that turns a Linear issue into a reviewable pull request, so Tezra ships the next product faster. _TypeScript_

**[important-things-mcp](https://github.com/tezra-io/important-things-mcp)** — give your agents a second brain. A local-first MCP server for important facts, citations, open loops, and captured inputs. _TypeScript_

**[RustyClaw](https://github.com/tezra-io/rustyclaw)** & **[zeroclaw](https://github.com/tezra-io/zeroclaw)** — runtime research on a Rust core: multi-agent architecture, local embeddings, and a tamper-proof memory ledger. _Rust_

**More:** [fermix-plugins](https://github.com/tezra-io/fermix-plugins) — integrations for Fermix · [usg](https://github.com/tezra-io/usg) — Claude Code / Codex usage from the CLI · [homebrew-tap](https://github.com/tezra-io/homebrew-tap) — install Fermix via Homebrew

### 🧭 how we work

Useful autonomy needs visible limits. Four rules we build by:

- **Local control** — keep the operator close to the machine, the files, and the service state.
- **Visible state** — write config, logs, traces, memory, and job history where you can inspect them.
- **Human checkpoints** — keep consequential changes behind explicit review, grants, or service commands.
- **Small systems** — prefer direct local components over remote bridges, hidden worker pools, and mystery state.
